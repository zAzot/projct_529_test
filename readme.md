## Розділ 2: Опис функціонала (ролі користувачів)

Система реалізує три рівні доступу. Кожен наступний рівень включає функціонал попереднього.

### 2.1. Клієнт (Гість / Користувач)
**Доступ:** без авторизації.  
**Основні можливості:**
- **Перегляд каталогу**: пагінований список товарів з таблиці `tbl_1_pub`.
- **Фільтрація та пошук**: за кодом, назвою, брендом, моделлю, місцем, кількістю, ціною, станом, датою публікації. Підтримується сортування.
- **Детальна картка**: перегляд усіх полів товару та його фотографій (з динамічним масштабуванням).
- **Інтерфейсна інформація**: отримання назви сайту (`/site-name`), логотипу (`/logo`), контактних даних (`/contacts`).

### 2.2. Адміністратор (admin)
**Доступ:** авторизація з роллю `admin`.  
**Основні можливості (всі функції клієнта +):**
- **Управління товарами (CRUD)**: 
  - Редагування будь-яких полів опублікованих товарів, включаючи текстові та числові (крім прямого редагування фото).
  - Заміна фотографії (старий файл автоматично видаляється з диска).
  - Видалення запису з будь-якої таблиці (опубліковані, неопубліковані, буфер, історія).
  - Переміщення між статусами "Опубліковано" та "Не опубліковано".
  - **Додавання нових товарів безпосередньо до "Неопублікованих"** (ендпоінт `/admin/table/unpublished/add`).
- **Управління продажами (Буфер/Історія)**:
  - **Старт переговорів**: списання товару зі складу (зменшення QTY) та створення запису у буфері.
  - **Завершення переговорів**: перенесення запису з буферу до історії з фіксацією дати завершення.
  - **Скасування переговорів**: повернення товару з буферу до публікації (з генерацією нового ID).
- **Масове завантаження**: імпорт каталогу з файлів `.xlsx` або `.db` із повною заміною даних.
- **Режим обслуговування**:
  - Вмикається **автоматично** під час завантаження файлу та вимикається після завершення операції.
  - Адміністратор має можливість **вимкнути** режим вручну через `/admin/maintenance-off` (наприклад, якщо операція зависла). Ручне ввімкнення не передбачено (лише через автоматичні процеси).

### 2.3. Суперадміністратор (super_admin)
**Доступ:** авторизація з роллю `super_admin` (вхід за логіном `none` та статичним кодом).  
**Основні можливості (всі функції адміністратора +):**
- **Управління адміністраторами**:
  - Реєстрація нових облікових записів (логін, email, пароль) з валідацією.
  - Видалення існуючих адміністраторів.
  - Перегляд списку всіх адмінів та їхніх ролей.
- **Система резервного копіювання**:
  - **Ручний бекап**: створення повної копії трьох основних таблиць у `manual_backup`.
  - **Відновлення**: вибіркове (за назвами таблиць) або повне відновлення з ручного або автоматичного (створеного перед завантаженням) бекапу.
- **Аудит та безпека**:
  - Перегляд **системного лог-файлу** (`logs/app.log`) через веб-інтерфейс з використанням статичного коду (захист від несанкціонованого доступу).
  - Перегляд повного **журналу дій адміністраторів** (`admin_logs`), де фіксується кожен важливий запит.
  - **Перевірка пароля** суперадміна через ендпоінт `/admin/verify-password` (використовується для підтвердження критичних операцій на клієнтській стороні).
- **Налаштування інтерфейсу**: оновлення контактних даних (телефону та імені), що відображаються на головній сторінці.

## Розділ 3: Технічний опис ендпоінтів та бази даних

Розділ для розробників та DevOps-інженерів. Містить вичерпну інформацію про структуру БД, всі доступні API-маршрути, механізми автентифікації та захисту.

### 3.1. Детальна схема бази даних (SQLite)

| Назва таблиці | Призначення | Ключові поля та примітки |
| :--- | :--- | :--- |
| **admins** | Облікові записи | `id` (PK), `username` (UNIQUE), `email` (UNIQUE), `password_hash` (формат: `salt$hash`, PBKDF2), `role` (admin/super_admin), `created_at` |
| **tbl_1_pub** | Опубліковані товари | `id` (PK), `Code`, `QTY`, `NAME`, `Brand`, `Model`, `PLACE`, `Price`, `Conditions`, `Truck_models`, `Publishing_date`, `Description`, `Photos` (рядок з назвами файлів через кому) |
| **tbl_2_unpub** | Неопубліковані товари | Аналогічна структурі `tbl_1_pub` |
| **tbl_3_buffer** | Буфер продажів | Додатково: `sales_negotiation_start_date` (TIMESTAMP DEFAULT CURRENT_TIMESTAMP) |
| **tbl_4_history** | Історія продажів | Додатково: `sales_negotiation_end_date` (TIMESTAMP) |
| **backup_table** | Автоматичний бекап | `id` (PK), `tbl` (джерело), `original_id`, + усі поля товарів. Створюється перед `/upload`. |
| **manual_backup** | Ручний бекап | Ідентична `backup_table`. Створюється суперадміном. |
| **admin_logs** | Журнал дій | `id`, `login`, `endpoint`, `request_data`, `response_data`, `created_at` |
| **db_version** | Версія схеми | `id`, `version`, `applied_at`. Використовується для міграцій. |
| **salt_table** | Службова | (Не використовується активно, але створюється для сумісності). |

**Важливо:** При вставці нових записів функція `get_all_existing_ids` перевіряє зайнятість ID у **всіх чотирьох** основних таблицях (`tbl_1_pub`, `tbl_2_unpub`, `tbl_3_buffer`, `tbl_4_history`) для уникнення конфліктів.

### 3.2. Механізми автентифікації та безпеки

- **JWT (Bearer Token)**: Основний спосіб авторизації. Токен передається у заголовку `Authorization: Bearer <token>`.
- **HttpOnly Cookie**: Після входу через `/token` токен додатково встановлюється у захищену cookie `access_token` для автоматичної передачі браузером.
- **WebSocket**: Авторизація відбувається через query-параметр `?token=<jwt_token>` при підключенні до `/ws/progress`.
- **Захист маршрутів**:
  - **`protected_routes_middleware`**: перехоплює запити до `/admin` та `/sierra-alpha` і перевіряє наявність валідного токена з ролями `admin` або `super_admin` відповідно.
  - **Залежності FastAPI**: ендпоінти використовують `Depends(get_current_admin)` або `Depends(get_current_super_admin)` для тонкого контролю доступу.

### 3.4. Приклад складного запиту (FilterRequest)
`POST /data/filter` приймає JSON-об'єкт:
```json
{
  "code": "ABC",
  "brand": "Toyota",
  "price_min": 100,
  "price_max": 500,
  "sort_by": "Price",
  "sort_order": "desc",
  "page": 1,
  "page_size": 20
}
```
Відповідь містить `total`, `page`, `page_size`, `total_pages` та масив `data` з об'єктами товарів (включаючи `photo_urls`).

### 3.5. Middleware захисту (як працює)
1. **`rate_limit_middleware`**: обмеження 100 запитів за 60 секунд з одного IP.
2. **`body_size_middleware`**: відхилення запитів > 10 МБ.
3. **`maintenance_middleware`**: якщо `MAINTENANCE_MODE=True`, перенаправляє всі запити, окрім білого списку (`/healthcheck`, `/ws/progress`, `/progress-status`, `/admin/maintenance-off`, `/admin/backup/*` тощо).
4. **`protected_routes_middleware`**: перевіряє JWT для `/admin` та `/sierra-alpha`. Перенаправляє на `/login` при невалідному або відсутньому токені.

### 3.6. Обробка фотографій
- При запиті фото через `/photos` або `/item` зображення стискаються до 480x480 (або кастомного `size`) за допомогою бібліотеки Pillow (LANCZOS).
- При старті сервера виконується "сміттяр": всі файли в папці `Photos/`, на які немає посилань у полях `Photos` таблиць БД, видаляються (крім `no_photo.png`).
